## Decisions Made

- **Mention handles resolve to the local part of a user's email** (before `@`), matched
  case-insensitively against workspace members. The schema has no dedicated `username`/`handle`
  field, and email is globally unique, so the email local-part is the most stable handle available
  in the MVP. Only members of the current workspace are notified.
- **Comment-notification recipients = issue assignees + creator.** TZ §5.18 says "comment on
  watched issue", but there is no watch/subscription model in the MVP; the people already involved
  with the issue stand in for watchers.
- **Mentions outrank the generic comment notification** so a mentioned, also-assigned user gets a
  single `mentioned` row rather than two notifications.
- **Mark-read on a non-owned notification returns 404** (not 403) so the API never confirms the
  existence of another user's notifications.
